# UI 发布质量门禁与灰度准入（Cursor Style）

本清单用于 UI 优化版本发布前验收，覆盖聊天区、任务面板、任务详情、通知中心四个关键模块。

## 1. 联动一致性门禁
- [ ] 调度触发后，聊天区出现事件卡片（不重复）。
- [ ] 调度触发后，任务面板出现历史记录（分组一致）。
- [ ] 调度触发后，通知中心出现 task 通知（不重复）。
- [ ] 聊天区点击“打开任务详情”可打开编辑区任务 Tab。
- [ ] 聊天区点击“打开关联对话”可切换到正确线程。
- [ ] 任务面板点击调度记录与聊天区主操作结果一致。

## 2. 交互可达性门禁
- [ ] 事件行支持鼠标点击与键盘 Enter/Space。
- [ ] 三点菜单可通过键盘打开并执行动作。
- [ ] 关键按钮具备 `aria-label` 或可读文本。
- [ ] 焦点样式可见（focus ring 不丢失）。

## 3. 视觉一致性门禁
- [ ] 聊天区/任务面板/任务详情使用一致的时间分组规则（刚刚/15分钟内/更早）。
- [ ] 状态色和状态文案一致（running/completed/failed/paused/cancelled）。
- [ ] 主信息与次信息分层清晰（摘要默认可见，细节按需展开）。

## 4. 数据与可观测门禁
- [ ] `/autonomous/schedule-state` 支持分页与过滤参数并返回稳定结构。
- [ ] 前端 API 兼容旧参数调用（无参数调用不报错）。
- [ ] 任务详情能看到与当前任务相关的调度事件时间线。
- [ ] 调度事件 key 去重生效（短窗内不重复通知/渲染）。

## 5. 稳定性门禁
- [ ] `npm run build` 通过。
- [ ] 关键改动文件 `ReadLints` 无新增问题。
- [ ] 后端 `py_compile` 通过（涉及改动文件）。
- [ ] 不出现事件风暴（持续重复跳转/重复通知）。

## 6. 灰度策略
- Stage 1：仅 personal scope 开启（默认）。
- Stage 2：观察 24h（重复通知率、错误日志、用户可用性反馈）。
- Stage 3：放开到 org/public。
- 回滚条件：任一核心门禁失败或出现高频重复联动。

## 7. 发布前检查（与 P1 可靠性联动）
- 发布前除本门禁 1–5 外，建议同步执行 [P1 可靠性改进清单](p1_reliability_improvement_backlog_2026-03-02.md)：success_rate/blocked_recovery_rate/deliverable_effective_rate 目标与发布门证据归档。
- 每次发布宜保留「自动化证据 + 人工签字 + 风险说明」三件套（见 P1-04）。
