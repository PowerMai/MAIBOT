# 云端模型配置说明

## 推荐方式：云端端点 + 动态发现（零配置模型列表）

只需配置「云端端点」的 Base URL 和 API Key，系统会自动拉取该端点下**所有**模型并加入可用列表，无需逐个添加。

**重要**：API Key 属于**端点**，该端点下所有发现的模型共用同一个 Key；配置页按端点展示「本端点有哪些模型、Key 是否已配置」。

### 配置步骤

1. **API Key 二选一**：
   - **环境变量**（推荐，不持久化明文）：`export CLOUD_QWEN_API_KEY=sk-你的Key`，在端点中填 `api_key_env: "CLOUD_QWEN_API_KEY"`。
   - **内联 Key**：在端点的「API Key / 环境变量」栏直接填写 `sk-` 开头的 Key（会写入 `models.json`，仅本地）。

2. **配置云端端点**（任选其一）：
   - **设置面板**：打开 设置 → 通用 → **云端端点（动态发现）** →「+ 添加端点」，填写 Base URL（如 `http://39.170.37.25:63000/v1`）和 API Key（环境变量名或 sk- 直填），保存后会自动刷新并发现该端点下所有模型；每个端点行会显示「Key：已配置/未配置」和「发现 N 个模型」。
   - **配置文件**：在 `backend/config/models.json` 中增加顶层键 `cloud_endpoints`：
     ```json
     "cloud_endpoints": [
       { "base_url": "http://39.170.37.25:63000/v1", "api_key_env": "CLOUD_QWEN_API_KEY" }
     ]
     ```
     其中 `api_key_env` 可为环境变量名或直接写 `sk-xxx`。重启后端或点击设置中的「刷新云端模型」后，发现的模型会出现在可用模型列表中。

3. **使用**：在聊天或设置里将主模型/子代理模型选为任意已发现的云端模型即可。窗口等参数会从端点返回的模型元数据中自动解析（若 API 提供 `context_length` / `max_model_len` 等）。

### 行为说明

- **Base URL**：保存时统一去掉末尾斜杠（如 `http://host/v1/` 存为 `http://host/v1`），与业界常见用法一致，前后端匹配一致。
- 启动时后台会拉取每个 `cloud_endpoints` 的 `GET /v1/models`，将返回的 `data[].id` 作为可用模型展示。
- 同一端点下的多个模型会全部出现在列表中，且共用该端点的 API Key；设置页按端点展示模型列表与 Key 状态。
- 设置中的「刷新云端模型」会重新拉取并更新列表；新增/修改端点后也会自动刷新并清空 LLM 缓存，新 Key 立即生效。

---

## 备选：预置单条云端模型（models.json）

若不想用动态发现，可在 `models.json` 的 `models` 数组中预置一条云端模型（如 `cloud/39-qwen`），并设置 `url`、`tier: "cloud"`、`api_key_env`。使用前需设置对应环境变量；模型 ID 需与端点 `GET /v1/models` 返回的 id 一致，否则在设置中修改该条目的「模型 ID」。

---

## 通过设置面板手动添加单个云端模型

1. 打开 **设置 → 模型**，点击「+ 添加模型」。
2. 填写 **模型 ID**（需与目标端点 `GET /v1/models` 返回的 id 一致）、**显示名称**。
3. **Provider** 选 **OpenAI**，**类型** 选 **云端 (cloud)**。
4. **API URL** 填 Base URL，**API Key 环境变量** 填变量名并在启动前 `export`。

添加失败时请检查：模型 ID 是否与已有条目重复；Provider 是否为白名单内之一。

### 某模型在列表中但选择后无响应或报 400（如 qwen3-coder-next）

- **ID 一致性**：请求上游时使用 **GET /v1/models 返回的原始 id**（`runtime_model_id`），不做改写。若仍报 400「资源包不支持该模型」，多为 **上游对 model 参数大小写或格式敏感**；后端已做兼容：对 `cloud/xxx` 形式会按「后缀 + 大小写不敏感」匹配到发现列表中的模型，并以其 `runtime_model_id` 请求上游，避免前后端大小写不一致导致误判。
- 列表中的展示 id 为「cloud/」+ 接口返回的 id；选择后发送的 model 与列表一致即可，后端会解析并带上正确的 `runtime_model_id`。
- 若仍不可用：在设置中「刷新云端模型」后查看该模型的实际 ID，或查看后端日志中「创建 LLM」行的 `request_model=` 确认发往上游的 id。

### 云端模型上下文窗口（如 262144）

- 本系统会从 GET /v1/models 的 `data[]` 中解析 `context_length` / `max_model_len` / `max_tokens` / `max_input_tokens`，未返回时云端默认按 262144 处理，以充分利用大窗口模型（如 qwen3.5）。
- 解析到的 `context_length` 会用于：1) 模型配置与前端展示；2) DeepAgent 的 `max_input_tokens`（压缩与上下文预算）。可据此充分利用 262144 等大窗口。

### 报错「资源包不支持该模型」（400）

该提示**来自云端服务商**（如阿里云百炼 / DashScope），表示当前账号或已购资源包未开通该模型。若你确认该模型已开通，多半是 **请求中的 model 与上游要求不一致**（如大小写）：后端已对云端模型 id 做兼容查找并统一使用发现时的 `runtime_model_id` 请求上游，可查看日志确认实际请求的 model。若仍报错，请在对应云端控制台核对模型 id 与开通情况。
