# 技能体系规划

基于市场优质 skills 与 L1–L5 体系化架构的规划文档，用于先规划、再接入，把市场与互联网资源用好。

## 一、市场 12 条与层级映射

以 `backend/config/skills_market.json` 的 12 条为基准：

| 市场 id | 名称 | domain | 规划层级 | requires_tier |
|---------|------|--------|----------|---------------|
| skill-knowledge-building-basic | 知识构建助手（基础版） | knowledge | L4 | free |
| skill-ontology-management-core | 本体管理工作流（Core） | knowledge | L4 | free |
| skill-document-generation-pro | 专业文档生成（Pro） | general | L3 | pro |
| skill-report-generation-pro | 报告生成与复盘（Pro） | general | L3 | pro |
| skill-data-analysis-pro | 数据分析增强（Pro） | analytics | L3/L4 | pro |
| skill-bidding-analysis-enterprise | 招标文件深度分析（Enterprise） | sales | L5 | enterprise |
| skill-proposal-writing-enterprise | 投标方案撰写（Enterprise） | sales | L5 | enterprise |
| skill-compliance-check-pro | 合规审查助手（Pro） | contract | L3/L5 | pro |
| skill-project-management-basic | 项目任务编排（基础版） | general | L2 | free |
| skill-workflow-design-pro | 流程编排设计（Pro） | workflow | L4 | pro |

L5 两条（招标分析、投标撰写）不单独暴露，纳入招投标专项能力包（bidding）。

## 二、目标技能清单（每层由谁填充）

- **L1**：无市场对应 → 内置 reasoning、ask-methodology（2 个）。
- **L2**：市场 skill-project-management-basic + 内置 plan、debug、review、self-learning、self-evolution、auto-discovery（6 个）。
- **L3**：市场 5 条（文档、报告、数据、合规、可视化）+ 内置 pdf、docx、xlsx、pptx（4 个）。
- **L4**：市场 4 条（知识构建、本体、流程编排、外部资源）+ 内置 skill-creator、mcp-builder、kb-*（7 个）。
- **L5**：仅 **bidding** 一个专项（内置 `knowledge_base/skills/bidding/SKILL.md`；插件 plugins/sales 为可选实现细节）。

## 三、角色暴露清单（约 10–20 个/角色）

| 角色 | 建议暴露 | 数量 |
|------|----------|------|
| 通用 Agent | L1 全 2 + L2 全 6 + L3 市场 5 + 格式 4 + L4 精选 4–6 | 约 20 内 |
| 解决方案专家 | L1 全 2 + L2 核心 3 + L3 格式 4 + 市场文档/报告/合规 3 + **bidding 1** | 约 13 |
| 知识工程师 | L1 全 2 + L2 核心 2 + L3 格式 2 + 市场知识/本体/外部资源 3 + 内置 kb-* 7 | 约 16 |
| 分析师 | L1 全 2 + L3 市场 5 + 内置格式 4 | 约 11 |
| 办公助手 | L1 全 2 + L3 市场 3 + 内置格式 4 | 约 9 |

## 四、专业使用 vs 一般使用

- **一般使用（free）**：`allow_skills` 仅限 platform、foundation、general、format、modes、community；`max_custom_skills: 5`。L1/L2/L3 基座多数免费可用。
- **专业使用（pro/enterprise）**：`allow_skills: ["*"]`，更高 max_custom_skills。L4 部分、L5 招投标可标 `requires_tier: "pro"` 或 `"enterprise"`。
- **落地**：list_skills/match_skills 与 BUNDLE 只暴露当前 tier 允许的技能；安装时 `tier_service.ensure_skill_install_allowed` 校验 requires_tier。规划与 `license_tiers.json`、`skills_market.json` 的 requires_tier 保持一致。

## 五、互联网 skills 资源与落地步骤

- **来源**：Playbooks.com、[Anthropic 官方 skills](https://github.com/anthropics/skills)、[Agent Skills 规范与生态](https://agentskills.io)、第三方市场；本系统支持 `skills_market.json` 的 `source_type: "remote"` + `remote_url` 拉取远程列表。
- **遴选**：按 L1–L5 对应、与现有条目重复度、描述与版权筛选；产出互联网技能资源分析表（附录或独立文档）。
- **落地**：配置 remote_url；安装走 POST /skills/install 并校验 requires_tier；按 §10.1–10.3 映射到层级与角色暴露清单。

## 六、单会话技能数上限

- 单会话暴露技能数可由 `skill_profiles.json` 的 `max_skills_per_session`（全局或 per-profile）限制，与 Claude 的「每请求技能数」理念一致，避免上下文过长；未配置或≤0 则不截断。`build_runtime_index` 在按 profile/tier 过滤后会按该上限截断（按 level 与 name 排序保留前 N 个）。

## 七、落地步骤摘要

1. 产出本规划文档（本文档）并维护与实现对齐。
2. 市场技能通过 POST /skills/install 或手工按层级落地 SKILL.md，并纳入 skill_profiles 的 paths。
3. 为 general、bidding、knowledge_engineering、analyst、office 等 profile 配置 paths，使 list_skills 仅返回该 profile 下技能（约 10–20 个/角色）。
4. BUNDLE 内联仅上述暴露清单的 name+description。
5. capability_registry 与角色 capabilities 与真实 SKILL 名或市场 id 一一对应；解决方案专家仅绑定「bidding」。
