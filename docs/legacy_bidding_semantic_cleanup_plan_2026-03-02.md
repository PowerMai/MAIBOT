# 历史 bidding 语义清理计划（2026-03-02）

## 背景

- 运行链路已迁移到 `sales` 语义，但文档/知识资产仍存在历史 `bidding`/`招标` 文案残留。
- 目标不是“一刀切删除”，而是“分层替换 + 可追溯迁移”，避免误伤真实业务语义。

## 原则

- 代码契约优先：影响运行时语义的配置/接口字段优先清理。
- 文档分层：规范文档先清、历史记录可保留但需标注“历史语义”。
- 全程可追溯：每次清理保留扫描报告与变更记录。

## 执行步骤

1. 基线扫描  
   - 执行：`backend/scripts/scan_legacy_bidding_terms.py`
   - 输出：`backend/data/legacy_bidding_scan_report.json`
2. 分类处理  
   - `运行时配置/接口`：必须替换为新语义；
   - `文档模板`：优先替换；
   - `历史报告`：可保留但增加“历史语义说明”。
3. 回归验证  
   - `check:single-agent` + `check:role-mode-contract` + `test_single_agent_api_acceptance`
4. 发布留痕  
   - 在发布签字中附带扫描报告与本轮清理范围。

## 通过标准

- 严格目标：运行时代码与配置命中为 0。
- 软目标：文档命中持续下降，且历史保留项均有注释说明。

## 建议节奏

- 每周一次非 strict 扫描（观察趋势）。
- 发布前一次 strict 扫描（针对“运行时目录”可单独设阈值 0）。
