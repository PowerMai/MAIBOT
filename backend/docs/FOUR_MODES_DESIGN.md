# 五种模式深度设计

## 核心理念

五种模式是**操作员控制台**的核心功能，帮助用户在不同阶段使用不同的交互方式：

```
用户工作流程：
Ask（咨询探讨）→ Plan（规划计划）→ Agent（自动执行）→ Review（结构化评审）→ Debug（问题诊断）
     ↑                                                                         │
     └─────────────────────────────────────────────────────────────────────────┘
```

## 一、模式定位与业务场景

### Agent 模式 - 自动执行者

**定位**：用户已经明确知道要做什么，让 AI 自主完成

**业务场景**：
- 文档业务：生成招标文件、合同、报告
- 教育业务：批改作业、生成教案、分析学情
- 制造业务：分析生产数据、生成报表
- 代码业务：编写代码、重构、测试

**特点**：
- 自主决策，遇到问题先尝试解决
- 完整的工具集（读写文件、执行代码、搜索等）
- 输出是可交付物（文件、代码、报告）
- 过程信息简化，聚焦结果

**上下文**：任务描述 + 相关文件路径 + Skill 指南
**工具**：全部可用
**输出**：执行结果 + 生成的文件

---

### Ask 模式 - 专业顾问

**定位**：用户不确定，需要探讨、分析、获取建议

**业务场景**：
- 文档业务：分析招标文件要点、评估投标策略
- 教育业务：讨论教学方法、分析学生问题
- 制造业务：诊断生产问题原因、评估方案
- 代码业务：讨论架构设计、评审代码

**特点**：
- 对话式交互，多问多答
- 可以分析项目文件（只读）
- 给出多角度分析和建议
- 不执行修改操作

**上下文**：对话历史 + 项目结构 + 相关知识
**工具**：只读工具（read_file, grep, search_knowledge, glob）
**输出**：分析报告 + 建议列表 + 后续问题

---

### Plan 模式 - 战略参谋

**定位**：长周期任务，需要详细规划和准备

**业务场景**：
- 文档业务：投标项目计划、文档编制计划
- 教育业务：学期教学计划、课程设计
- 制造业务：生产计划、项目排期
- 代码业务：重构计划、迁移方案

**特点**：
- 深度分析任务的各个方面
- 生成结构化的执行计划
- 识别风险和依赖关系
- 输出是计划文档（可保存供 Agent 执行）；用户确认后计划可直接作为可用执行子代理（或主线程）的输入

**计划结构**（与 mode_config、[SUBAGENT_OUTPUT_SPEC.md](SUBAGENT_OUTPUT_SPEC.md) 一致）：呈现计划时必须包含目标、交付物（路径+类型+验收）、步骤（至少 id/action/output_path）、风险与假设、需用户确认的点。

**上下文**：任务描述 + 资源清单 + 约束条件 + 历史计划
**工具**：分析工具 + write_file（保存计划）+ write_todos
**输出**：执行计划 + 里程碑 + 风险评估 + 资源需求

**Plan 模式：确认时机与实现位置**  
「先出计划再确认」由**图与前端**协作完成，不依赖 human_in_the_loop 中间件：main_graph 根据 configurable 中的 `plan_confirmed`、`plan_phase` 决定当前为 planning 还是 execution 阶段；前端在用户确认后下发 `plan_confirmed=true` 或 `plan_phase=execution`。故 middleware_chain.json 中 plan 链未包含 human_in_the_loop。详见 main_pipeline_and_middleware_rationality.md。

---

### Debug 模式 - 问题诊断专家

**定位**：出错或结果不符合预期，需要追踪和诊断

**业务场景**：
- 文档业务：为什么文档内容不对？检查生成过程
- 教育业务：为什么评分不准确？检查评分逻辑
- 制造业务：为什么数据分析结果异常？检查数据源
- 代码业务：为什么代码不工作？检查执行过程
- 工具业务：为什么 MCP 工具调用失败？检查连接和参数
- 外部资源：为什么数据库查询失败？检查连接和 SQL

**特点**：
- 收集和分析过程信息（日志、中间结果）
- 逐步追踪问题根源
- 对比预期和实际结果
- 提供修复建议

**上下文**：错误信息 + 执行日志 + 中间结果 + 配置信息
**工具**：诊断工具 + 日志查看 + 状态检查 + 重放执行
**输出**：诊断报告 + 根因分析 + 修复方案 + 验证步骤

---

### Review 模式 - 结构化评审员

**定位**：需要质量审查、合规核对或方案评估，输出结构化评审结论

**业务场景**：
- 文档业务：评审投标文件完整性、合规性与风险项
- 教育业务：评审课程方案可执行性与教学目标对齐度
- 制造业务：评审生产分析报告结论是否有证据支撑
- 代码业务：评审需求实现是否满足规范与验收标准

**特点**：
- 清单驱动审查，逐项给出证据与结论
- 支持交叉验证（文件、知识库、联网信息）
- 保持原始业务文件只读
- 使用 `write_file` 输出结构化评审报告

**上下文**：评审目标 + 检查清单 + 证据来源
**工具**：只读分析 + `write_file`（仅报告输出）
**输出**：评审报告 + 严重级别分布 + 优先修复建议

---

## 二、模式差异对比

| 维度 | Agent | Ask | Plan | Review | Debug |
|------|-------|-----|------|--------|-------|
| **目标** | 完成任务 | 获取建议 | 制定计划 | 结构化评审 | 定位问题 |
| **交互风格** | 自主执行 | 对话探讨 | 结构化输出 | 清单审查 | 追踪分析 |
| **执行操作** | ✅ 完整 | ❌ 只读 | ⚠️ 保存计划 | ⚠️ 仅输出评审报告 | ⚠️ 诊断操作 |
| **工具范围** | 全部 | 只读 | 分析+写计划 | 只读+写报告 | 诊断+检查 |
| **上下文重点** | 任务+资源 | 对话+知识 | 目标+约束 | 清单+证据 | 日志+过程 |
| **输出形式** | 文件/结果 | 建议/分析 | 计划文档 | 评审报告 | 诊断报告 |
| **Temperature** | 0.3 | 0.7 | 0.5 | 0.3 | 0.2 |

## 三、过程信息记录与读取

### 执行日志 API

- **写入**：Agent 执行工具链时由 main_graph 的 deepagent_node 调用 execution_logger 的 start_task/complete_task 等，将执行步骤与结果写入 execution_log。
- **读取**：Debug 模式通过 `GET /execution-logs?thread_id=...&limit=...&status=...` 拉取指定 thread 的最近执行记录（steps、final_result、error 等），用于分析问题根因。

### 过程信息记录系统

Debug 模式需要丰富的过程信息，因此需要在 Agent 执行时记录：

```python
# 执行日志结构
execution_log = {
    "task_id": "xxx",
    "mode": "agent",
    "start_time": "2025-01-22T10:00:00",
    "steps": [
        {
            "step_id": 1,
            "action": "read_file",
            "input": {"path": "招标文件.pdf"},
            "output": {"content": "...", "size": 12345},
            "duration_ms": 150,
            "success": True,
        },
        {
            "step_id": 2,
            "action": "task",
            "input": {"agent": "<available-planning-subagent>", "description": "..."},
            "output": {"plan": "..."},
            "duration_ms": 3500,
            "success": True,
            "llm_calls": [
                {"prompt_tokens": 1200, "completion_tokens": 500, "model": "qwen3"}
            ]
        },
        # ...
    ],
    "final_result": {...},
    "error": None,
}
```

- 显示执行进度条
- 工具调用折叠显示
- 最终结果突出
- 生成的文件列表

### Ask 模式 UI
- 对话式气泡

## 五、模式切换工作流

```
场景：投标文档生成

1. Ask 模式 - 咨询阶段
   用户："这个招标文件的关键要求是什么？"
   AI：分析招标文件，给出要点列表和建议

2. Plan 模式 - 规划阶段
   用户："帮我规划投标文档的编制计划"
   AI：生成详细的执行计划，保存到 outputs/投标计划.md

3. Agent 模式 - 执行阶段
   用户："按照计划开始编制投标文档"
   AI：自动执行计划，生成各章节内容

4. Review 模式 - 评审阶段
   用户："先审查这版方案是否满足招标要求并给出风险分级"
   AI：按清单逐项核对并输出结构化评审报告

5. Debug 模式 - 诊断阶段
   用户："技术方案章节内容不对，帮我诊断原因"
   AI：检查执行日志，分析哪个步骤出错，给出修复方案
```

## 六、实现要点

1. **动态提示词**：根据模式生成不同的系统提示词
2. **工具过滤**：根据模式限制可用工具
3. **上下文组装**：根据模式选择不同的上下文内容
4. **过程记录**：Agent 执行时记录详细日志供 Debug 使用
5. **UI 适配**：前端根据模式渲染不同的组件
6. **完成语义单源**：各模式的「完成条件」在 `mode_config.py` 中集中定义：`MODE_COMPLETION_MARKERS`（Plan/Debug/Review 校验用最小关键词集）与 `MODE_COMPLETION_FAIL_SUGGESTION`（未通过时的建议文案），由 `DoneVerifier` 单源引用，与 `get_mode_prompt()` 中 `<mode_completion_criteria>` 文案对齐，避免提示词与校验双源不同步。

## 七、Plan 确认双入口

计划确认提供两种等价入口，产品上以「有中断时以 InterruptDialog/内联中断区为主；计划卡上『确认执行』为同态操作」为约定：

- **中断区**：当存在 `plan_confirmation` 中断时，用户在聊天区底部内联区点「批准」→ `resumeInterrupt(threadId, 'approve')`；若配置项 `maibot_plan_confirm_switch_to_agent` 为 true（默认），批准成功后前端切换为 Agent 模式并派发 `CHAT_MODE_CHANGED`，与 PlanToolUI 一致。
- **计划卡片**：在计划工具卡上点「确认执行」时，若当前有 plan_confirmation 中断则同样 `resumeInterrupt` 并切 Agent；若无中断则派发 `PLAN_CONFIRMED`，由 thread 发确认消息，是否切 Agent 由同一配置项决定。

两处均读取同一配置，保证行为一致。详见 `docs/ux_and_ops_notes.md` §9。
