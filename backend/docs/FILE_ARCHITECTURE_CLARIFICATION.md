# 文件架构说明

## 核心问题解答

### Q: 用户所有内容都在前端本地文件夹中吗？

**不完全是**。架构如下：

```
┌─────────────────────────────────────────────────────────────────┐
│                         前端 (Electron)                          │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │ 用户本地文件夹（用户指定）                                     │ │
│  │ - /Users/xxx/Documents/项目A/                                │ │
│  │ - 用户上传的文件                                              │ │
│  │ - 用户生成的输出                                              │ │
│  └─────────────────────────────────────────────────────────────┘ │
│                              ↓ MCP Client                        │
└─────────────────────────────────────────────────────────────────┘
                               │
                               ↓ HTTP/WebSocket
┌─────────────────────────────────────────────────────────────────┐
│                    后端 (LangGraph Server)                       │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │ 1. 知识库 (PROJECT_ROOT/knowledge_base/)                     │ │
│  │    - skills/: SKILL.md 文件                                  │ │
│  │    - global/domain/: 领域知识                                │ │
│  │    - 向量索引: data/vectorstore/                             │ │
│  ├─────────────────────────────────────────────────────────────┤ │
│  │ 2. 数据持久化 (PROJECT_ROOT/data/)                           │ │
│  │    - checkpoints.db: 会话状态                                │ │
│  │    - store.db: 长期记忆                                      │ │
│  ├─────────────────────────────────────────────────────────────┤ │
│  │ 3. 临时工作区 (PROJECT_ROOT/backend/tmp/)                    │ │
│  │    - 默认工作空间（用户未指定时）                             │ │
│  │    - 缓存文件                                                │ │
│  └─────────────────────────────────────────────────────────────┘ │
│                              ↓ MCP Server (可选)                 │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │ 4. MCP 代理到前端（云端部署模式）                             │ │
│  │    - 文件操作代理到用户本地                                   │ │
│  └─────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
```

### Q: 后端 read/write 都是通过 MCP 操作前端？

**取决于部署模式**：

1. **LOCAL 模式（当前）**：
   - 后端直接操作本地文件系统
   - FilesystemMiddleware 直接读写 `PROJECT_ROOT/knowledge_base/` 和 `backend/tmp/`
   - 用户工作区通过路径配置指定

2. **CLOUD 模式（未来）**：
   - 后端通过 MCP Server 代理文件操作
   - MCP Client 在用户本地执行实际操作
   - 适合云端部署场景

### Q: 后端除了知识库文件夹之外，还有其他文件夹吗？

**是的**：

| 文件夹 | 位置 | 用途 | 持久化 |
|--------|------|------|--------|
| knowledge_base | PROJECT_ROOT/ | Skills、领域知识 | ✅ |
| data | PROJECT_ROOT/ | Checkpointer、Store、向量索引 | ✅ |
| backend/tmp | backend/ | 临时工作区、缓存 | ❌ |
| outputs | 用户指定 | 生成的文件 | ✅ |

## 附件处理与文件夹的关系

### 当前实现

```python
# attachment_processor.py
# 附件处理流程：
# 1. 前端上传文件 → 临时目录 /tmp/attachments/
# 2. 预处理（解析、摘要）
# 3. 如果需要保存 → 用户工作区 outputs/
# 4. 返回处理结果（摘要 + 文件路径）
```

### 应该统一的地方

1. **上传文件路径**：应该使用 DeepAgent 的 FilesystemMiddleware 配置
2. **工作区文件夹**：应该与 `backend/tmp/workspace/` 统一
3. **知识库文件**：只读，不应该通过附件上传修改

## API 端点简化

### Q: Claude/Cursor 也是一大堆 API 端点吗？

**不是**。Claude/Cursor 的核心 API 非常简洁：

```
POST /chat          # 发送消息（核心）
GET  /threads       # 获取会话列表
POST /threads       # 创建会话
DELETE /threads/:id # 删除会话
```

其他功能通过 **消息的 additional_kwargs** 传递：

```json
{
  "message": "分析这个文件",
  "additional_kwargs": {
    "mode": "agent",
    "attachments": [{"type": "file", "path": "..."}],
    "web_search": true,
    "context_items": [...]
  }
}
```

### 建议简化

将 `/context/*` 的功能合并到主 API：

```
POST /chat                    # 核心聊天 API
  - 自动计算 token
  - 自动处理附件
  - 自动管理上下文

GET  /chat/state/:thread_id   # 获取会话状态（包含 token 统计）
POST /chat/compress/:thread_id # 手动触发压缩
```

## 模式与 SubAgent 的关系

### 当前设计问题

Ask 模式应该能访问整个项目（像 Agent 一样），但当前设计将其限制为只用知识类工具。

### 正确的设计

```
┌─────────────────────────────────────────────────────────────────┐
│                     Orchestrator                                 │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │ 模式判断：根据 mode 配置决定行为                              │ │
│  │                                                              │ │
│  │ Agent: 全流程（理解→规划→执行→输出）                         │ │
│  │ Plan:  理解→规划→输出（跳过执行）                            │ │
│  │ Ask:   理解→分析→输出（可用所有分析工具，不执行修改操作）      │ │
│  │ Debug: 理解→诊断→修复→验证                                   │ │
│  └─────────────────────────────────────────────────────────────┘ │
│                              ↓                                   │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │ SubAgents（所有模式都可以使用，但行为不同）                   │ │
│  │                                                              │ │
│  │ planning-subagent（可用时）:                                 │ │
│  │   - Agent/Debug: 完整规划 + 执行步骤                         │ │
│  │   - Plan: 详细规划 + 方案对比（不执行）                       │ │
│  │   - Ask: 快速分析 + 建议                                     │ │
│  │                                                              │ │
│  │ execution-subagent（可用时）:                                │ │
│  │   - Agent/Debug: 执行工具调用                                │ │
│  │   - Plan: 不调用（跳过）                                     │ │
│  │   - Ask: 只读操作（分析、搜索，不修改）                       │ │
│  │                                                              │ │
│  │ knowledge-subagent（可用时）:                                │ │
│  │   - 所有模式: 知识检索、技能查询                             │ │
│  └─────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
```

### 关键区别

| 模式 | Planning | Executor | Knowledge | 输出特点 |
|------|----------|----------|-----------|----------|
| Agent | 完整规划 | 完整执行 | 检索 | 执行结果 |
| Plan | 详细规划 + 对比 | 跳过 | 检索 | 规划文档 + 思维导图 |
| Ask | 快速分析 | 只读分析 | 检索 | 对话式建议 |
| Debug | 诊断规划 | 修复执行 | 检索 | 调试报告 |

## 实现建议

1. **统一文件路径**：使用 DeepAgent 的 FilesystemMiddleware 配置
2. **简化 API**：合并到主聊天 API
3. **模式控制**：通过 Orchestrator 提示词控制 SubAgent 行为
4. **附件处理**：不需要额外模型，使用 LangChain 文档加载器
